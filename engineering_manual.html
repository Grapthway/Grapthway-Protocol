What is Grapthway?

Grapthway is a decentralized API access layer — a programmable gateway network where no single entity owns or controls the routing infrastructure. It unifies multiple downstream GraphQL and REST microservices behind a single, composable API surface while running on a peer-to-peer network with no central database, no central broker, and no single point of failure.

Think of it as a "decentralized API Gateway" — similar in function to Kong or AWS API Gateway, but fully distributed. Any node in the network can receive a client request. Any node can resolve schemas and route traffic. Any node can coordinate long-running background jobs. If a node goes offline, the network automatically heals.

Decentralized Routing
All service configs, schemas, and state are stored in a distributed Kademlia DHT, propagated via GossipSub. No config servers needed.
Unified API Surface
Merges GraphQL schemas from all registered services into a single stitched schema. REST services are proxied transparently.
Cryptographic Identity
Every participant — node operators, developers, end-users — has a secp256k1 keypair. All actions are signed and verified on-chain.

Core Concepts

Before diving into setup, internalize these five concepts. They are referenced throughout the entire manual.

  • Node — A running Grapthway process. Nodes form the P2P network. Each node has a cryptographic identity (keypair) that proves it is the legitimate operator.
  • Server Node — A node with GRAPTHWAY_ROLE=server. Exposes an HTTP API to clients. Receives requests, runs pipelines, and coordinates durable workflows.
  • Worker Node — A node with GRAPTHWAY_ROLE=worker. No public API. Listens for durable workflow tasks on the P2P bus and executes them.
  • Service — A downstream GraphQL or REST microservice registered by a developer. The gateway proxies traffic to it.
  • Pipeline — A declarative JSON workflow that runs before (pre) or after (post) a main service call. Enables auth checks, enrichment, fan-out, and rollback logic.
  • Durable Workflow — An asynchronous, long-running pipeline whose state is checkpointed in the DHT. Survives node crashes and retries failed steps automatically.
  • GCU — Grapthway Compute Unit. The native token used to pay for API request processing. 1 GCU = 1,000,000 micro-GCU internally.
  • DHT — Distributed Hash Table (Kademlia). Used to store service configs, workflow state, and schema CIDs without a central database.
  • GossipSub — The pub/sub messaging layer. Nodes gossip config updates, hardware stats, and network state to all peers in real-time.
  • CID — Content Identifier. A hash-based key used in the DHT. A config's CID changes when its contents change, enabling integrity verification.

P2P Architecture

Grapthway is built entirely on libp2p. There is no central server, no managed database, and no message broker. The network is self-organizing and self-healing.

Core P2P Components

  • Kademlia DHT — All service configurations, workflow state checkpoints, and schema CIDs are stored in this distributed hash table. Any node can read or write data. Data is replicated across multiple peers for resilience.
  • GossipSub — The pub/sub bus. Used to propagate config update notifications (ConfigUpdateTopic), hardware stats, network state, and workflow task announcements to all connected peers in real-time without flooding.
  • libp2p Streams — Direct peer-to-peer TCP connections for low-latency coordination, such as a worker claiming a specific workflow task from a server coordinator.
  • Badger v4 — An embedded key-value store used as the local node's persistent state. It backs the DHT and the ledger, ensuring data survives restarts.

Node Roles

A Grapthway node runs in one of two roles, set via the GRAPTHWAY_ROLE environment variable. The role fundamentally changes what the node does.

Server Node (server)
  • Exposes the full HTTP/HTTPS API to clients
  • Serves the unified GraphQL endpoint
  • Acts as coordinator for durable workflows
  • Runs synchronous orchestration pipelines
  • Announces itself to the network via SELF_URL
Worker Node (worker)
  • No public HTTP API exposed
  • Listens for workflow task announcements on GossipSub
  • Executes durable workflow steps in isolation
  • Checkpoints workflow state to the DHT
  • Can recover orphaned workflows from crashed coordinators

High Availability & Fault Tolerance

The network is designed to tolerate node failures without data loss or service interruption.

Durable Workflow Recovery Protocol

When a server node (coordinator) goes offline mid-workflow, its in-flight workflows become "orphaned." The recovery protocol is fully automatic:

Detection
A worker attempts to claim a task via a libp2p stream to the original coordinator. The stream fails to open, indicating the coordinator is offline.
State Retrieval
The worker queries the DHT using the workflow instance ID to retrieve the last saved checkpoint state.
Recovery Proposal
The worker broadcasts a RecoveryProposal message on GossipSub. It includes the last known state and nominates a new coordinator.
Consensus & Takeover
The first available server node that processes the proposal takes ownership. It updates the DHT with its own coordinator ID and announces the next task step.
Zero Manual InterventionThis entire recovery process is automatic. Mission-critical workflows like order fulfillment continue without any operator action when a server node restarts or crashes.

Kubernetes Guide

The recommended production deployment. Use separate Deployments for server and worker pools, an Ingress for TLS, and NetworkPolicies to lock down downstream services.

Key Best Practices

  • Separate Deployments — Run server nodes and worker nodes in separate Kubernetes Deployment objects. This lets you scale each pool independently (e.g., auto-scale workers during peak background job loads).
  • NGINX Ingress for TLS — Grapthway is an application gateway, not a network ingress. Use NGINX Ingress Controller or Traefik to handle TLS termination and route HTTPS:443 to Grapthway's HTTP:5000.
  • NetworkPolicy for Security — Lock downstream microservices so they are only accessible from Grapthway pods. This is a critical security hardening step.
  • Secrets for Private Keys — Store NODE_OPERATOR_PRIVATE_KEY in a Kubernetes Secret, not in a ConfigMap or plain environment variable.
  • Headless Service for P2P — Workers need stable DNS for P2P discovery. Use a Kubernetes Headless Service (clusterIP: None) for the worker pods.

Security & Wallets

Cryptographic Identity

Grapthway uses secp256k1 elliptic curve cryptography — the same curve used by Bitcoin and Ethereum. Every identity consists of:

  • Private Key — A 256-bit random number in hex. Used to sign requests. Never transmitted. Must be kept secret.
  • Public Key — The 65-byte uncompressed public key derived from the private key. Prefixed 04. Can be shared freely.
  • Address — A 20-byte (40 hex chars) identifier prefixed with 0x. Derived by SHA-256 hashing the public key and taking the last 40 hex chars. This is the canonical identity used in all transactions.

Admin Authentication (Node Operators)

Admin routes (/admin/*) require signature verification. The node's identity is set by NODE_OPERATOR_PRIVATE_KEY at startup.

What to sign for Admin requests: The node operator's own public wallet address string (e.g. "0x1a2b3c..."). Hash it with SHA-256, then sign the hash with secp256k1 DER encoding. Send as hex in X-Grapthway-Admin-Signature.
Required admin headers 
X-Grapthway-Admin-Address: 0x1a2b3c...    # Your node's wallet address
X-Grapthway-Admin-Signature: 3045...       # Hex DER-encoded signature of SHA256(address)

User Authentication (Developers & End-Users)

Protected API routes (/api/v1/*) use a different signing scheme that proves ownership without exposing private keys.

What to sign for User requests: For GET requests: the full request URI string. For POST requests: the raw JSON request body bytes. Hash with SHA-256, sign with secp256k1 DER.
Required user auth headers 
X-Grapthway-User-Address: 0xabcdef...         # Your wallet address
X-Grapthway-User-PublicKey: 04abcdef...        # Your full uncompressed public key (hex)
X-Grapthway-User-Signature: 3045...            # Hex DER-encoded signature
Never commit private keys to source control. Use environment variables, Docker secrets, or a secrets manager (Vault, AWS Secrets Manager, etc.). A leaked private key gives full control of the corresponding wallet and any funds inside it.

For Engineers & Developers
Building services, defining orchestration, and publishing to the Grapthway network

Overview & Developer Identity

As a developer, you build standard GraphQL or REST microservices in any language. You do not need to modify them. You register them on Grapthway by publishing a SignedServiceConfig JSON object that describes the service's schema and orchestration logic.

Getting a Developer Wallet

Ask any Node Operator to run POST /admin/wallet/create on their node. They'll give you your address, publicKey, and privateKey. Guard the private key — it is your identity on the network.

Funding Your Account

All API requests passing through Grapthway debit Grapthway Compute Units (GCU) from your developer account. You must maintain a positive GCU balance. Ask a node operator to credit your address or transfer GCU from another address using POST /api/v1/wallet/transfer.

Developer Penalty BoxIf your GCU balance is insufficient when a request arrives, the gateway rejects the request with a 402 Payment Required error. Repeated failures may trigger a penalty box that temporarily suspends your services from the network until you top up your balance.

Full Client Code (grapthway-client.js)

grapthway-client.js 
import 'dotenv/config';
import elliptic from 'elliptic';
import { createHash } from 'crypto';
import fetch from 'node-fetch';

const EC = elliptic.ec;
const ec = new EC('secp256k1');

export class GrapthwayClient {
    constructor(privateKeyHex) {
        if (!privateKeyHex) throw new Error('Private key is required.');
        this.keyPair = ec.keyFromPrivate(privateKeyHex, 'hex');
        this.publicKeyHex = this.keyPair.getPublic('hex');
        const pubKeyBytes = Buffer.from(this.publicKeyHex.substring(2), 'hex');
        const pubKeyHash = createHash('sha256').update(pubKeyBytes).digest();
        this.address = `0x${pubKeyHash.toString('hex').slice(-40)}`;
    }

    /**
     * Signs a service configuration object.
     * The config is canonicalized (keys sorted), then SHA-256 hashed and signed.
     * Returns a new object with all config fields plus a `signature` field.
     */
    signConfig(config) {
        // Build canonical config — only include known fields, strip nulls
        let canonical = {
            developerPubKey: this.publicKeyHex,
            middlewareMap: config.middlewareMap,
            path: config.path,
            restPipelines: config.restPipelines,
            schema: config.schema,
            service: config.service,
            stitchingConfig: config.stitchingConfig,
            type: config.type,
        };
        Object.keys(canonical).forEach(k => {
            if (canonical[k] === null || canonical[k] === undefined) delete canonical[k];
        });

        // Sort keys deterministically
        const sortedKeys = Object.keys(canonical).sort();
        const canonicalJSON = `{${sortedKeys.map(k => `"${k}":${JSON.stringify(canonical[k])}`).join(',')}}`;

        const hash = createHash('sha256').update(canonicalJSON, 'utf8').digest();
        const signature = this.keyPair.sign(hash);
        const derSignature = signature.toDER('hex');

        const signed = {};
        sortedKeys.forEach(k => { signed[k] = canonical[k]; });
        signed.signature = derSignature;

        return signed;
    }

    /**
     * Publishes a signed config to a Grapthway node.
     * Requires GRAPTHWAY_NODE_URL environment variable.
     */
    async publish(signedConfig) {
        const nodeUrl = process.env.GRAPTHWAY_NODE_URL;
        if (!nodeUrl) throw new Error('GRAPTHWAY_NODE_URL is not set.');

        const body = JSON.stringify(signedConfig);
        const requestHash = createHash('sha256').update(body, 'utf8').digest();
        const requestSignature = this.keyPair.sign(requestHash).toDER('hex');

        const res = await fetch(`${nodeUrl}/api/v1/publish-config`, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'X-Grapthway-Developer-ID': this.address,
                'X-Grapthway-Config-Signature': requestSignature,
                // User auth headers (also required by UserAuthMiddleware)
                'X-Grapthway-User-Address': this.address,
                'X-Grapthway-User-PublicKey': this.publicKeyHex,
                'X-Grapthway-User-Signature': requestSignature,
            },
            body,
        });

        if (!res.ok) {
            throw new Error(`Publish failed: ${res.status} ${await res.text()}`);
        }
        console.log(`✅ Published config for: ${signedConfig.service}`);
    }

    /**
     * Sends periodic heartbeats to keep the service registration live.
     * Call this every 30 seconds from a running service instance.
     */
    async heartbeat(nodeUrl, service, serviceUrl, type, subgraph, configCid, path) {
        const body = JSON.stringify({
            service, url: serviceUrl, type, subgraph,
            configCid, path, developerPubKey: this.publicKeyHex,
        });
        const hash = createHash('sha256').update(body, 'utf8').digest();
        const sig = this.keyPair.sign(hash).toDER('hex');

        await fetch(`${nodeUrl}/api/v1/health`, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'X-Grapthway-Developer-ID': this.address,
                'X-Grapthway-Config-Signature': sig,
                'X-Grapthway-User-Address': this.address,
                'X-Grapthway-User-PublicKey': this.publicKeyHex,
                'X-Grapthway-User-Signature': sig,
            },
            body,
        });
    }
}

export const grapthwayClient = new GrapthwayClient(process.env.SERVICE_PRIVATE_KEY);

/**
 * Express middleware to verify incoming requests are from a Grapthway node.
 * Add to your downstream services to ensure only Grapthway can call them.
 */
export const grapthwayVerificationMiddleware = (req, res, next) => {
    const signature = req.headers['x-grapthway-signature'];
    const nodeAddress = req.headers['x-grapthway-node-address'];
    if (!signature || !nodeAddress) return res.status(403).send('Forbidden: Missing Grapthway signature.');
    next(); // Optionally verify the signature cryptographically here
};

Orchestration Pipelines

Pipelines are the core of Grapthway's synchronous orchestration. They are declarative JSON arrays of steps that execute before (pre) and after (post) your main service call. They work for both GraphQL fields and REST routes, and can freely mix GraphQL and REST calls within a single flow.

Pipeline Context

A shared JSON object (the "context") is created at request start and passed through every pipeline step. Steps read from it and write to it using assign.

Pre-Pipeline Initial Context
GraphQL: All request arguments are under args. Access with "args.userId", "args.email", etc.

REST: JSON request body is under request.body. Access with "request.body.amount", etc.
Post-Pipeline Merged Context
After your main service responds, the context is merged with all top-level fields from the service's JSON response. So you can use both initial args and the service result in post-steps.

operationstring(GraphQL only) "query" or "mutation". Defaults to "query".
methodstring(REST only) HTTP method: "GET", "POST", "PUT", "PATCH", "DELETE".

Transactional Pipelines & Rollbacks

When a blocking step in a pre pipeline fails, Grapthway can automatically execute compensating "rollback" actions for previously successful steps — in reverse order.

Rollback Execution OrderRollbacks are triggered in reverse step order. If Step 1 and Step 2 succeed but Step 3 fails, Step 2's rollback runs first, then Step 1's rollback — like unwinding a stack.
"createUser": {
  "pre": [
    {
      "service": "user-db",
      "field": "insertUser",
      "argsMapping": { "details": "args.newUser" },
      "assign": { "newUser": "" },       // Save created user to context
      "onError": {
        "stop": true,
        "rollback": [
          {
            "service": "user-db",
            "field": "deleteUserById",
            "argsMapping": { "id": "newUser.id" }   // Uses context saved by assign
          }
        ]
      }
    },
    {
      "service": "email-api",
      "method": "POST",
      "path": "/send-welcome",
      "bodyMapping": { "email": "newUser.email" },
      "onError": { "stop": true }    // Failure here triggers user-db rollback above
    }
  ]
}

Durable Workflows

Durable Workflows are long-running, asynchronous business processes that are fully decoupled from client request lifecycles. They can run for minutes, hours, or days, surviving network failures and node restarts through state checkpointing in the DHT.

Durable vs. In-Memory

In-Memory Workflows
  • Config: Default (isDurable: false)
  • Runs in the server node's memory
  • Not guaranteed — lost if node restarts
  • retryPolicy has no effect
  • Best for: lightweight, non-critical fire-and-forget tasks
Durable Workflows
  • Config: Requires isDurable: true
  • Asynchronous — hands off to worker nodes via P2P
  • Guaranteed — state persisted in DHT
  • retryPolicy fully supported per step
  • Best for: mission-critical processes (orders, billing, compliance)
Advanced Pattern: Reliable In-Memory via Durable ParentYou can make an in-memory workflow reliable by placing it inside a durable workflow's callWorkflow step with a retryPolicy. If the inner (in-memory) workflow fails, the durable worker retries the entire inner workflow from scratch. This achieves reliability without the overhead of full durable state tracking for every sub-step.

Workflow Config Flags

FlagTypeDescription
isWorkflowbooleanMarks this pipeline config as a reusable workflow. Can be called via callWorkflow from other pipelines.
isDurablebooleanIf true, uses the durable engine: P2P task distribution, DHT state persistence, retries. Requires isWorkflow: true.
isInternalbooleanIf true, cannot be attached directly to a GraphQL field or REST route. Can only be triggered via callWorkflow.

Schema Stitching

Schema stitching allows you to automatically extend a type from one service with fields resolved from another service — without changing either service's code. The gateway handles cross-service data fetching transparently.

How It Works

Define a stitchingConfig in your service config. When the gateway resolves a field of that type, it automatically calls the resolver service with the specified argument mapping, then merges the result into the parent object.

// In the "products-service" config:
"stitchingConfig": {
  "Product": {                              // Type to extend
    "inventoryStatus": {                    // New field to add
      "service": "inventory-service",       // Resolver service
      "resolverField": "getInventoryById",  // GraphQL field to call
      "argsMapping": { "productId": "id" }  // Map parent's "id" → resolver's "productId"
    },
    "reviews": {
      "service": "review-service",
      "resolverField": "getReviewsByProduct",
      "argsMapping": { "productId": "id" }
    }
  }
}

Now when a client queries { getProduct(id: "123") { id, name, inventoryStatus { qty }, reviews { rating } } }, Grapthway automatically fans out to the inventory and review services and stitches the results back into a single response.

Security Best Practices for Developers

  • Sign All Downstream Requests — Use the grapthwayVerificationMiddleware in your downstream services to verify that requests are coming from a trusted Grapthway node.
  • Use NetworkPolicy (K8s) — Ensure your downstream services cannot be directly accessed by any caller except Grapthway pods.
  • Always Use onError: { stop: true } on Auth Steps — Any pipeline step that performs authentication or authorization must have stop: true to ensure the gateway halts on failure.
  • Use passHeaders for JWT Tokens — Pass the original Authorization header to downstream services that need to verify user sessions.
  • Do Not Store Secrets in Config — The SignedServiceConfig is public on the DHT. Never put API keys, database passwords, or other secrets in it.
  • Rotate Your Developer Key — If your private key is compromised, publish a new config with a new developerPubKey immediately.
The Economic Layer
GCU tokens, staking, custom tokens, allowances, and fee architecture

GCU Token

The Grapthway Compute Unit (GCU) is the native token of the network. All API request processing fees are paid in GCU. Internally, the ledger operates in micro-GCU (1 GCU = 1,000,000 µGCU) for precision.

Debit (Fee)
Each API request through the gateway debits a small GCU fee from the developer's account. Fees compensate node operators for compute resources.
Transfer
GCU can be freely transferred between addresses using the signed transfer endpoint. Supports nonce-based replay protection.
Reward
Node operators earn GCU rewards for processing requests and running the network infrastructure.

Staking System

Token holders can stake GCU to support the network and earn rewards. Staking has four operations: deposit, assign to a node, unassign from a node, and withdraw.

OperationDescriptionConstraints
Stake DepositMove GCU from your liquid balance into your stake pool (unassigned stake).Amount must be ≤ available balance
Stake AssignAllocate a portion of your stake pool to a specific node (by peer ID). Signals support for that node and enables reward flow.NodePeerID must be a valid libp2p peer ID
Stake UnassignRemove stake from a node. Funds enter an unbonding period before becoming withdrawable. The target node must be offline to unassign.Target node must not be currently connected to the P2P network
Stake WithdrawMove GCU from your unassigned (or post-unbonding) stake pool back to liquid balance.Unbonding period must have elapsed
Unbonding PeriodWhen you unassign stake from a node, it enters an unbonding period before you can withdraw it. This protects against rapid stake manipulation. The exact unbonding duration is set by the network protocol parameters.

Allowance System (GCU)

The allowance system lets a token owner pre-authorize a third party ("spender") to spend up to a specified GCU amount on their behalf. This is critical for pipeline-level automated payments (e.g. a pipeline step that pays a fee from a user's wallet).

Use Case ExampleA developer can set an allowance for the Grapthway protocol address, allowing the system to automatically debit API usage fees without requiring a new signed transaction for each request.

Fee Architecture

Every request that passes through the Grapthway gateway incurs a small GCU fee. The fee system works as follows:

  • Fee Calculation — Fees are calculated based on request complexity and the pipeline steps executed. The FeeInfo context (with DeveloperID, NodeOperatorID, and Cost) is attached to every request via the WithEconomic middleware chain.
  • Fee Flow — The developer is debited. A portion goes to the node operator as a processing reward. The LedgerAggregator batches these micro-transactions (TxBatchSize = 10,000 or every 10 seconds) into efficient batch aggregation blocks.
  • Penalty Box — If a developer's balance is too low, requests fail with a fee error. The DeveloperPenaltyBox tracks developers with repeated fee failures and can temporarily suspend their service registrations from being routed to.
  • Rollback Transactions — If a pipeline step fails after a fee has been charged, the ledger emits a ROLLBACK_CREDIT transaction to refund the developer. This ensures atomic fee semantics.

These are the routes through which end-users interact with the stitched API. The gateway routes, authenticates via the economic middleware, and proxies to the appropriate downstream service.

The primary GraphQL endpoint. Executes queries and mutations against the stitched schema for the specified developer's subgraph. The economic middleware debits the appropriate GCU fee from the developer's account before routing.
